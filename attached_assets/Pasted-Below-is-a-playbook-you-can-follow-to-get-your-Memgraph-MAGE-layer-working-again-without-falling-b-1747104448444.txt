Below is a playbook you can follow to get your Memgraph + MAGE layer working again without falling back to in-memory storage.
In short, the failures you saw come from two separate but related issues:

Cypher grammar – Memgraph does not allow a WHERE clause to be placed immediately after YIELD. You must pass the yielded variables through a WITH (or RETURN … WHERE …) first. 
Memgraph
Out-of-date procedure names – since Memgraph 3.0 the Neo4j-style CALL db.index.vector.queryNodes( … ) API was replaced by vector_search.search() (and related helpers). 
Memgraph
Everything else—driver, connection parameters, MAGE loading—works once those two points are fixed.

1 Fixing the YIELD … WHERE pattern

1.1 Listing loaded procedures
/*  before (throws) */
CALL mg.procedures() YIELD name WHERE name CONTAINS "vector" RETURN name;

/*  after (Memgraph-compliant) */
CALL mg.procedures() YIELD name
WITH name          /* <= funnels the stream */
WHERE name CONTAINS "vector"
RETURN name;
The Memgraph docs show the canonical form CALL mg.procedures() YIELD * without any inline WHERE; filtering is always demonstrated in a later clause. 
Memgraph
Memgraph

1.2 Filtering vector-search results
/*  BEFORE – fails: WHERE right after YIELD  */
CALL vector_search.search("index_docs", 20, $embedding)
YIELD node, similarity
WHERE similarity >= 0.65
RETURN node, similarity;

/*  AFTER – works on Memgraph 3.x */
CALL vector_search.search("index_docs", 20, $embedding)
YIELD node, similarity
WITH node, similarity
WHERE similarity >= 0.65
RETURN node, similarity;
2 Migrating from db.index.vector.queryNodes to vector_search.search

The old Neo4j syntax you copied (db.index.vector.queryNodes) is not shipped in Memgraph 3.x.
Memgraph’s own docs and every current tutorial now use the vector_search module. 
Memgraph
Memgraph
If you really need a minimum-similarity cutoff you pass it as the second argument limit or filter afterwards, as in §1.2.
A quick “hello-world”:

CREATE VECTOR INDEX index_docs ON :Document(embedding)
       WITH CONFIG {"dimension": 768, "capacity": 100_000};

CALL vector_search.search("index_docs", 5, $queryEmbedding)
YIELD node, similarity
WITH node, similarity
WHERE similarity > 0.7
RETURN node.title, similarity
ORDER BY similarity DESC;
The full flow (index creation ➜ search) appears in Memgraph’s vector-search guide. 
Memgraph

3 Loading (or re-loading) MAGE modules

If you run the official memgraph/memgraph-mage:latest Docker image the vector-search module is pre-installed but not automatically loaded every time:

CALL mg.load_all();               /* load every query module */  -- good for dev
-- or
CALL mg.load("vector_search");    /* load exactly what you need */
mg.procedures() will confirm that vector_search.* procedures are available.

4 Node.js client-side changes

Nothing is wrong with the driver; just point it at the database that actually has the module loaded:

import { createClient } from "@memgraph/bolt";

const mg = createClient({
  host: process.env.MG_HOST ?? "localhost",
  port: 7687,              // default Bolt port
  username: "memgraph",
  password: "memgraph"
});
Memgraph’s Node.js quick-start still shows the same connection string (bolt://localhost:7687).
Memgraph

5 Concrete patches for your codebase

file	change
memgraphClient.fixed.v2.ts	replace every ... YIELD ... WHERE ... with the two-step YIELD … WITH … WHERE … form shown above.
mageVectorService.ts	swap db.index.vector.queryNodes(...) for vector_search.search(...) with identical parameter order; keep the WITH … WHERE filter.
memgraphDiagnostic.js	add one test case per fixed query and assert that summary.counters().containsUpdates() is false (read-only) and records.length > 0.
Running your diagnostic script after these edits should print ✅ for every query; failures will most likely be due to a missing module load.

7 Why these changes matter

Memgraph 3.x refocused its Cypher grammar to stay close to openCypher but not to the APOC/Neo4j extensions—that’s why the old vector-index procedure vanished.
